QQ 聊天代理设计文档 v1.002

Abstract

本文档定义 QQ 聊天代理的一期 MVP 设计。系统运行在 Codex 中，面向单一用户，所有数据仅保存在本地，先接入 QQ，支持 $chat 本人直发和 $qchat 代理聊天两种入口，并在高风险或不确定场景下进入暂停确认态。

1. 设计目标

一期要解决的是一条可用的最小闭环：

  1. 通过别名找到联系人
  2. 读取最近会话上下文
  3. 按命令执行直发或代理回复
  4. 在本地保存聊天记录、画像、标签和决策摘要
  5. 在不确定时暂停并询问用户，而不是乱发或报错

一期不追求完全自治，也不追求复杂前端。

2. 非目标

以下内容不属于一期目标：

  • 微信接入
  • 多平台联系人统一管理
  • 长时间无人监管的自动闲聊
  • 多轮自治策略优化
  • 复杂参数系统
  • 云同步、多人共享、服务端存储

4. 联系人模型

4.1 别名原则

系统不直接依赖 QQ 昵称。联系人由用户维护的别名表解析。

目标：

  • 提高匹配稳定性
  • 避免昵称变化导致失效
  • 为后续画像和标签绑定提供固定锚点

4.2 contacts.yaml

建议文件路径：

建议结构：

字段说明：

  • contacts.<key>: 内部稳定标识
  • aliases: 用户输入可匹配的别名集合
  • qq: QQ 联系人唯一标识
  • display_name: 默认展示名
  • tags: 静态标签
  • notes: 用户备注

规则：

  • 一个 alias 在全局必须唯一
  • 匹配到多个联系人时直接报歧义，不做猜测
  • 未命中时返回明确错误

6. 系统模块划分

6.1 Alias Resolver

职责：

  • 读取 contacts.yaml
  • 做 alias 到 qq 的解析
  • 检查歧义和不存在情况

输入：alias 输出：contact_id, qq, display_name

6.2 QQ Adapter

职责：

  • 发送消息
  • 拉取最近消息
  • 接收新消息事件

要求：

  • 封装具体第三方接入细节
  • 对上层暴露统一接口
  • 失败时给出明确错误

6.3 Context Loader

职责：

  • 读取最近 N 条会话
  • 读取 profile 和动态标签
  • 读取最近决策摘要
  • 组装成代理可用上下文

6.4 Agent Engine

职责：

  • 处理 $qchat
  • 按任务目标生成回复
  • 判断是否可直接发送
  • 决定进入发送态还是暂停态

6.5 Message Sender

职责：

  • 执行统一发送逻辑
  • $chat / $qchat 共用
  • 发送后写聊天记录

6.6 Memory Store

职责：

  • 读写 chats / profiles / decisions / sessions
  • 保证结构稳定
  • 提供最小一致性约束

8. 风险控制

8.1 系统级硬约束

一期固定启用：

  • 不侮辱用户
  • 不损害用户名誉
  • 不擅自作出高风险承诺

8.2 必须暂停的情况

  • 系统对上下文理解不确定
  • 无法判断对方情绪
  • 涉及承诺
  • 涉及金钱
  • 涉及见面安排
  • 涉及争吵升级
  • 可能损害用户形象或名誉
  • 联系人标签和当前策略冲突

8.3 可直接发送的情况

  • 低风险日常闲聊
  • 问答明确且上下文充分
  • 任务目标简单
  • 联系人画像和动态标签支持当前表达方向

9. 标签更新策略

标签支持两种来源：

  • 系统自动生成
  • 用户手动补充或修正

一期原则：

  • 系统可写入建议型动态标签
  • 命中高不确定度时应先询问再落标签
  • 静态规则标签优先级高于动态标签

示例：

  • 静态：不能太主动
  • 动态：最近在生气

10. 实现顺序

建议按以下顺序推进：

  1. 建立本地数据目录和 schema
  2. 实现 contacts.yaml 与 alias resolver
  3. 实现 QQ adapter 最小收发接口
  4. 打通 $chat 全链路
  5. 实现 context loader
  6. 实现 $qchat 基础生成与发送
  7. 实现暂停确认态
  8. 实现决策摘要与动态标签更新

这样做的原因：

  • $chat 是最短闭环，先验证接入层可靠性
  • $qchat 依赖上下文和风控，适合第二阶段接入
  • 暂停态是代理模式可用性的底线，不应该最后补

11. 成功标准

MVP 达标条件：

  • $chat 能稳定通过别名发送消息
  • $qchat 能读取最近上下文并生成合理回复
  • 高风险或不确定时不会乱发，而会暂停询问
  • 所有数据仅保存在本地
  • 修改 contacts.yaml 和 profile 文件后，系统行为能随之变化

12. 已知风险

  • QQ 接入层依赖非官方生态，稳定性存在不确定性
  • 动态标签如果更新过度，可能学偏
  • 暂停态如果设计太频繁，会影响使用流畅度
  • 只靠最近上下文，可能无法覆盖长期关系细节

13. 下一步

正式设计之后，下一步不再补概念，而是进入原型实现：

  1. 创建项目内原型目录
  2. 落数据目录与 schema
  3. 先实现 $chat
  4. 再实现 $qchat 和暂停态

备注

本文档是正式设计版 v1.002，基于 2026-04-01 的需求确认整理，可作为原型开发基线。
