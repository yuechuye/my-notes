QQ 聊天代理 MVP

Abstract

目标是在 Codex 中做一个仅供我本人使用、数据只保存在本地、先接入 QQ 的聊天代理。它能够自动读取会话上下文，结合历史聊天记录、联系人画像和动态标签，按照我的高层指令生成并发送符合我交流习惯的回复；当判断不确定或命中风险边界时，必须停下来问我。

这份笔记是干什么的

这是一份 MVP 方向确认笔记，用来沉淀当前已经聊清楚的产品边界、交互方式、风险约束和一期实现重点。

它不是最终实现设计，但可以作为后续进入正式设计或原型开发前的参考底稿。

为什么做

这个项目要解决的不是“AI 会不会写一句回复”，而是下面几件事：

  • 遇到我不会回、懒得回、想回得更像我时，能直接代我处理
  • 对不同联系人，回复风格、关系边界、语气松弛度都不一样，需要区分
  • 在日常闲聊场景里，尽量减少我来回切换、构思和措辞的负担
  • 长期把聊天上下文和关系状态沉淀在本地，越用越像我

已确认的产品边界

用户与数据边界

  • 只给我自己使用，不做多用户系统
  • 聊天记录只保存在本地
  • 一期先做 QQ，后续效果好再考虑微信
  • 运行入口就在 Codex 中，不额外做复杂前端

自动化边界

  • 允许自动发送
  • 如果系统不确定，必须停下来问我
  • 一期目标偏日常闲聊，不只是重要对话辅助

安全边界

当前已明确的硬约束：

  • 不能生成侮辱我、影响我名誉的内容
  • 不能在高风险或高不确定度场景下擅自继续

后续可继续补充更多系统级 guardrail。

联系人与记忆设计

为什么不能只靠原始聊天记录

同一个人面对不同联系人，语气和策略会明显不同。

真正需要建模的是：

  • 我的全局表达风格
  • 我与每个联系人的关系风格
  • 当前会话状态
  • 最近上下文和未完成话题

所以数据层不应只保存原始聊天记录，而应拆成多层。

建议的数据层

  1. 原始聊天记录
  • 记录消息原文、时间、方向、发送状态等事实数据
  1. 联系人画像
  • 关系类型
  • 语气松弛度
  • 可否开玩笑
  • 是否适合暧昧表达
  • 常聊话题
  • 敏感点/禁区
  • 默认回复策略
  1. 动态标签
  • 允许系统自动生成
  • 系统不确定时可以询问我
  • 可以体现阶段性状态，例如“最近在生气”“最近关系升温”“这个话题要谨慎”
  1. 决策摘要
  • 每次发送前后的简短策略说明
  • 方便后续复盘为什么这样回复
  • 为后续优化画像提供依据

标签机制

标签需要支持动态更新，而不是只靠人工静态维护。

标签来源

  • 系统根据聊天内容自动推断
  • 用户手动补充或修正
  • 系统不确定时先询问我再落标签

标签分层建议

  1. 静态规则标签
  • 例如：不能开黄腔、不能太主动、可以开玩笑
  1. 动态状态标签
  • 例如：她在生气、最近聊得不错、刚发生误会
  1. 系统级硬约束
  • 不侮辱我
  • 不损害我名誉
  • 不擅自做超出边界的承诺

不要把这三类完全混在原始聊天文本里，最好单独存储。

自动发送与暂停规则

虽然允许自动发送，但一期必须是保守策略。

必须暂停并询问我的情况

  • 系统对上下文理解不确定
  • 涉及承诺
  • 涉及金钱
  • 涉及见面安排
  • 涉及争吵升级
  • 涉及可能损害我形象或名誉的表达
  • 联系人标签与当前生成策略冲突

可以直接发送的情况

  • 低风险日常闲聊
  • 明确问答且上下文清晰
  • 命令目标简单且无敏感边界
  • 联系人画像和动态标签都支持当前表达方向

为什么先做 QQ

结论是：QQ 比微信更适合做第一版接入。

原因：

  • 现有非官方协议生态比微信更现实
  • 更容易做本地监听和消息收发
  • 更适合先验证“像不像我”和“能不能闲聊”这两个核心价值

但需要明确：

  • 这依然不是零风险路径
  • 稳定性会依赖第三方接入层
  • 后续平台策略变化可能影响可用性

一期推荐范围

一期建议做成：单轮任务代理 + 可连续触发，而不是无限自主聊天。

一期要做

  • QQ 接入
  • 本地聊天记录存储
  • 联系人画像
  • 动态标签
  • $qchat 任务驱动命令
  • $qchat-send 直接代发命令
  • 自动发送前的风险判断
  • 不确定时停下来问我

一期先不做

  • 长时间无人监管的自主聊天
  • 无限多轮自动代聊
  • 无边界承诺、见面、金钱、敏感表态
  • 微信接入

当前最合理的一句话 MVP

Tip

一个仅供本人使用、数据只保存在本地、先接入 QQ 的 Codex 聊天代理。它可以自动读取上下文，接收类似 $qchat 用户名 任务 的指令，结合历史记录、联系人画像和动态标签生成并可直接发送符合本人风格的回复；在高风险或不确定场景下必须停下来询问本人。

为什么这样记更合适

相比直接保存原始对话，这种笔记结构更适合长期维护，因为它把内容分成了：

  • 已确认边界
  • 交互模型
  • 数据模型方向
  • 风险控制规则
  • 一期范围

这样后续无论是继续补设计，还是改成正式设计文档，都更容易复用。

待进入正式设计的问题

进入实现前，至少还要细化这些内容：

  • 用户名 的匹配规则：备注名、昵称还是别名映射
  • qchat 命令协议：参数格式、默认行为、错误处理
  • 标签的数据结构和更新规则
  • 风险分类与暂停阈值
  • 多轮状态如何保存
  • 决策摘要记录格式
  • 本地存储结构与加密策略

下一步

如果要继续推进，这份笔记之后最自然的后续是：

  • 进一步整理成正式设计文档
  • 拆出接入层、数据层、代理层的职责
  • 明确一期原型的命令协议和本地存储结构

备注

这份笔记基于 2026-04-01 的一轮需求澄清整理，属于 MVP 方向确认文档，不是最终实现设计。
