# QQ 聊天代理 MVP v1.001

> [!abstract]
> 目标是在 Codex 中做一个仅供我本人使用、数据只保存在本地、先接入 QQ 的聊天代理。它能够自动读取会话上下文，结合历史聊天记录、联系人画像和动态标签，按照我的高层指令生成并发送符合我交流习惯的回复；当判断不确定或命中风险边界时，必须停下来问我。

## 一句话定义

这是一个运行在 Codex 中的本地 QQ 聊天代理，命令入口固定为：

```text
$chat  <别名> <消息文本>
$qchat <别名> <任务目标>
```

其中：

- `$chat` 表示我本人聊天，系统只负责解析联系人、发送和记录
- `$qchat` 表示代理聊天，系统根据上下文、标签、画像和任务目标决定怎么说
- `$qchat` 内建暂停确认机制，不额外拆新命令

## 这份笔记是干什么的

这是一份 MVP 方向确认笔记，用来沉淀当前已经聊清楚的产品边界、交互方式、风险约束和一期实现重点。

它不是最终实现设计，但可以作为后续进入正式设计或原型开发前的参考底稿。

## 为什么做

这个项目要解决的不是“AI 会不会写一句回复”，而是下面几件事：

- 遇到我不会回、懒得回、想回得更像我时，能直接代我处理
- 对不同联系人，回复风格、关系边界、语气松弛度都不一样，需要区分
- 在日常闲聊场景里，尽量减少我来回切换、构思和措辞的负担
- 长期把聊天上下文和关系状态沉淀在本地，越用越像我

## 已确认的产品边界

### 用户与数据边界

- 只给我自己使用，不做多用户系统
- 聊天记录只保存在本地
- 一期先做 QQ，后续效果好再考虑微信
- 运行入口就在 Codex 中，不额外做复杂前端

### 自动化边界

- 允许自动发送
- 如果系统不确定，必须停下来问我
- 一期目标偏日常闲聊，不只是重要对话辅助

### 安全边界

当前已明确的硬约束：

- 不能生成侮辱我、影响我名誉的内容
- 不能在高风险或高不确定度场景下擅自继续

后续可继续补充更多系统级 guardrail。

## 交互方式

### 模式一：`$chat` 本人聊天

用户已经知道自己要说什么，系统只负责联系人解析、发送和落库。

示例：

```text
$chat 小雨 今天天气真好
$chat 老王 我刚忙完，晚点聊
```

特点：

- 没有代理决策
- 风险最低
- 是一条直接发送通道

### 模式二：`$qchat` 代理聊天

用户只给任务目标，系统结合上下文、关系状态和动态标签自行组织回复。

示例：

```text
$qchat 小雨 哄好她
$qchat 老王 回答他刚才的问题
$qchat 阿明 和他闲聊
$qchat 小李 自然结束对话
$qchat 小张 把话题转到周末安排
```

特点：

- 最符合真实使用习惯
- 用户不需要自己先写一版草稿
- 系统需要理解当前局面、关系状态和这次聊天目标

### 模式三：`$qchat` 暂停确认态

当系统判断不清楚时，不报错，也不硬发，而是进入暂停态追问最小必要信息。

示例：

```text
我不确定她现在是在生气还是在开玩笑。
建议先确认：要偏哄一点，还是先解释？
```

特点：

- 属于 `$qchat` 内建回退机制
- 目的是受控中断，而不是失败退出
- 确认后继续执行原任务

## 联系人与别名设计

联系人不直接靠 QQ 昵称匹配，而是由我先维护别名表。

例如：

- `小雨 -> 3391855334`
- `帅哥 -> 3391855334`

为了便于扩展，建议使用结构化文件而不是单行映射。

### 建议格式：`contacts.yaml`

```yaml
contacts:
  xiaoyu:
    aliases:
      - 小雨
      - 宝宝
    qq: "3391855334"
    tags:
      - 可安抚
      - 可闲聊
```

这样做的好处：

- 别名匹配稳定
- 后面能直接挂静态标签
- 以后扩微信时也不会推翻结构

## 联系人与记忆设计

### 为什么不能只靠原始聊天记录

同一个人面对不同联系人，语气和策略会明显不同。

真正需要建模的是：

- 我的全局表达风格
- 我与每个联系人的关系风格
- 当前会话状态
- 最近上下文和未完成话题

所以数据层不应只保存原始聊天记录，而应拆成多层。

### 建议的数据层

1. 原始聊天记录
- 记录消息原文、时间、方向、发送状态等事实数据

2. 联系人画像
- 关系类型
- 语气松弛度
- 可否开玩笑
- 是否适合暧昧表达
- 常聊话题
- 敏感点/禁区
- 默认回复策略

3. 动态标签
- 允许系统自动生成
- 系统不确定时可以询问我
- 可以体现阶段性状态，例如“最近在生气”“最近关系升温”“这个话题要谨慎”

4. 决策摘要
- 每次发送前后的简短策略说明
- 方便后续复盘为什么这样回复
- 为后续优化画像提供依据

## 标签机制

标签需要支持动态更新，而不是只靠人工静态维护。

### 标签来源

- 系统根据聊天内容自动推断
- 用户手动补充或修正
- 系统不确定时先询问我再落标签

### 标签分层建议

1. 静态规则标签
- 例如：不能开黄腔、不能太主动、可以开玩笑

2. 动态状态标签
- 例如：她在生气、最近聊得不错、刚发生误会

3. 系统级硬约束
- 不侮辱我
- 不损害我名誉
- 不擅自做超出边界的承诺

不要把这三类完全混在原始聊天文本里，最好单独存储。

## 自动发送与暂停规则

虽然允许自动发送，但一期必须是保守策略。

### `$chat` 的行为

- 不做代理推理
- 解析别名后直接发送
- 发送结果写入本地记录

### `$qchat` 的行为

- 先读取最近上下文
- 再读取联系人画像、标签和历史摘要
- 生成候选回复时必须先过风险判断
- 满足安全条件时直接发送
- 不满足时进入暂停确认态

### 必须暂停并询问我的情况

- 系统对上下文理解不确定
- 无法判断对方情绪
- 涉及承诺
- 涉及金钱
- 涉及见面安排
- 涉及争吵升级
- 涉及可能损害我形象或名誉的表达
- 联系人标签与当前生成策略冲突

### 可以直接发送的情况

- 低风险日常闲聊
- 明确问答且上下文清晰
- 命令目标简单且无敏感边界
- 联系人画像和动态标签都支持当前表达方向

## 为什么先做 QQ

结论是：QQ 比微信更适合做第一版接入。

原因：

- 现有非官方协议生态比微信更现实
- 更容易做本地监听和消息收发
- 更适合先验证“像不像我”和“能不能闲聊”这两个核心价值

但需要明确：

- 这依然不是零风险路径
- 稳定性会依赖第三方接入层
- 后续平台策略变化可能影响可用性

## 一期推荐范围

一期建议做成：单轮任务代理 + 可连续触发，而不是无限自主聊天。

### 一期要做

- QQ 接入
- `contacts.yaml` 别名表
- 本地聊天记录存储
- 联系人画像
- 动态标签
- `$chat` 本人聊天命令
- `$qchat` 代理聊天命令
- `$qchat` 暂停确认态
- 自动发送前的风险判断
- 不确定时停下来问我

### 一期先不做

- 长时间无人监管的自主聊天
- 无限多轮自动代聊
- `$chat` 附加复杂参数
- 多平台联系人映射
- 无边界承诺、见面、金钱、敏感表态
- 微信接入

## 当前最合理的一句话 MVP

> [!tip]
> 一个仅供本人使用、数据只保存在本地、先接入 QQ 的 Codex 聊天代理。它提供 `$chat` 和 `$qchat` 两种入口，前者用于本人直发，后者用于代理任务；系统通过 `contacts.yaml` 解析联系人别名，结合历史记录、联系人画像和动态标签进行生成，在高风险或不确定场景下进入暂停确认态。

## 为什么这样记更合适

相比直接保存原始对话，这种笔记结构更适合长期维护，因为它把内容分成了：

- 已确认边界
- 交互模型
- 联系人解析方式
- 数据模型方向
- 风险控制规则
- 一期范围

这样后续无论是继续补设计，还是改成正式设计文档，都更容易复用。
